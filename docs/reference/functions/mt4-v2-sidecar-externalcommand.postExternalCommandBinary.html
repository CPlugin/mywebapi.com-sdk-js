[**@mywebapi.com/sdk**](../README.md) *** [@mywebapi.com/sdk](../index.html) / [mt4-v2-sidecar-externalcommand](../modules/mt4-v2-sidecar-externalcommand.html) / postExternalCommandBinary # Function: postExternalCommandBinary() > **postExternalCommandBinary**(`tradePlatform`, `mT4ExternalCommandBinaryRequest?`, `options?`): `Promise`\<[`postExternalCommandBinaryResponseSuccess`](../types/mt4-v2-sidecar-externalcommand.postExternalCommandBinaryResponseSuccess.html)\> Defined in: mt4-v2-sidecar-externalcommand.ts:61 Pass arbitrary binary payload to the MT4 server's plugin pipeline.
Manager (live) call via the wrapper's `ExternalCommandCustom` overload with a pass-through byte-array marshaller. The first installed plugin that returns `RET_OK` wins; its response bytes become the payload's `Data` field.
Wire format: `byte[]` serializes as base64 in JSON. Clients agree with the plugin author on the binary layout.
Why sidecar-only: the wrapper's binary variants rely on `Marshal.SizeOf` + native `ExternalCommand` dispatch. On `mtmanapi64.dll` the signed/unsigned marshalling diverges from `mtmanapi.dll` — same payload byte-for-byte can decode differently on x64. The x86 sidecar process loads the 32-bit DLL where the marshalling is the original one.
The wrapper's third `ExternalCommandCustom` overload that takes an `ICustomSerializer` interface is genuinely not REST-translatable — it requires the caller to provide C# serialization logic in-process.
